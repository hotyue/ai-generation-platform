# AI Generation Platform
Security Whitepaper（安全白皮书）
## 1. 文档定位与法律地位

本文档用于定义 AI Generation Platform 在 v1.0.0-beta 阶段的安全模型、信任边界与防护策略。

### 1.1 法律地位（冻结）

- 本文档属于 生产级安全约束文档

- 对 代码实现、部署方式、运维操作 具有 强制约束力

- 后续版本（v1.0.1+）不得否定本文档已确认的安全事实

## 2. 安全设计总原则（冻结事实）
### 2.1 核心安全原则
| 原则	    |           说明              | 
| ----------- | --------------------------- |
| 最小权限	    |   用户只能访问其角色允许的资源 |
| 单一事实源	|  安全判断以数据库为准 |
| 显式边界	    |   每个组件的信任边界清晰 |
| 不信任客户端	|  客户端状态永不作为安全依据 |


## 3. 系统信任边界模型
### 3.1 信任分层（事实）
```scss
[ Browser / Client ]
        ↓  (不可信)
[ Frontend (Nginx + SPA) ]
        ↓  (受限信任)
[ Backend API (FastAPI) ]
        ↓  (强信任)
[ Database (PostgreSQL) ]
```
### 3.2 边界定义
层级	信任级别	说明
浏览器	❌ 不可信	所有输入均可伪造
前端	⚠️ 半可信	仅负责展示与转发
后端	✅ 可信	唯一业务裁决者
数据库	✅ 强可信	最终事实来源
## 4. 身份认证与会话安全
### 4.1 认证机制（冻结）

- 使用 JWT Bearer Token

- Token 仅由后端签发

- Token 存储于浏览器 localStorage

### 4.2 JWT 安全约束
项目	策略
签名密钥	服务端私有
校验位置	后端中间件
前端参与	❌ 不参与校验
Token 内容	不包含敏感信息
## 5. 权限模型（RBAC）
### 5.1 角色定义（冻结）
角色	权限范围
user	生成、历史、个人信息
admin	用户管理、套餐管理
### 5.2 权限裁决原则

- 权限由 后端强制校验

- 前端仅做 UI 控制（非安全手段）

- 所有敏感接口 必须依赖 get_current_user

## 6. API 安全策略
### 6.1 API 统一入口

- 所有 API 统一使用 /api 前缀

- WebSocket 使用 /api/ws

### 6.2 安全措施
项目	策略
CORS	明确允许来源
Authorization	强制 Bearer
参数校验	Pydantic
状态校验	以 DB 状态为准
## 7. 数据安全与隔离
### 7.1 用户数据隔离（冻结）

- 所有业务表 必须包含 user_id

- 所有查询 必须显式按 user_id 过滤

- 严禁 前端传入 user_id 作为裁决依据

### 7.2 关键数据表安全级别
表名	安全级别	说明
users	高	身份核心
quota_logs	高	财务 / 计费事实
histories	高	用户生成记录
plans	中	管理数据
## 8. 生成链路安全
### 8.1 生成流程事实

- 前端请求 /api/generate

- 后端鉴权 + quota 校验

- 后端调用 ComfyUI

- 写入 History（pending）

- 异步状态更新

### 8.2 安全保证点

- quota 扣减 先于结果

- History 记录 不可伪造

- 图片 URL 不直接暴露内部路径

## 9. 网络与部署安全
### 9.1 容器网络模型

- 使用 Docker bridge 网络

- backend / db 不直接暴露公网（推荐）

- frontend 为 唯一公网入口

### 9.2 外部依赖信任
服务	信任策略
ComfyUI	受限信任（仅 HTTP）
CDN	可替换，不影响核心
Cloudflare	可选加固层
## 10. 运维安全边界
### 10.1 严格禁止操作

- 直接修改生产数据库数据

- 在容器内手动 patch 代码

- 跳过 Git / 镜像版本

### 10.2 Secret 管理（当前阶段）

- JWT_SECRET 通过环境变量注入

- 不写入 Git

- 不写入镜像层

## 11. 审计与追溯能力
### 11.1 已具备能力（事实）

- quota_logs：计费审计

- histories：行为审计

- 用户登录日志（基础）

### 11.2 未纳入（后续）

- IP 级审计

- 行为风控

- 自动封禁策略

## 12. 威胁模型总结
威胁	应对策略
前端篡改	后端强校验
Token 伪造	JWT 签名
用户串号	user_id 强绑定
越权访问	RBAC
## 13. 冻结声明（v1.0.0-beta）

- 以下安全事实已冻结：

- 后端为唯一裁决者

- 数据库为最终事实源

- 前端不具备安全权威

- 所有用户隔离基于 user_id

## 14. 总结

安全不是“功能”，
而是 系统可信度的下限。

本安全模型为平台未来引入 支付、企业租户、多租户隔离 提供稳定基座。
